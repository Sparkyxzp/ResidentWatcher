# Software Requirements Specification (SRS)
**ชื่อโปรเจกต์:** DormBill (ระบบจัดการและตรวจสอบค่าเช่าหอพักออนไลน์)

## 1. บทนำ (Introduction)

### 1.1 จุดประสงค์ (Purpose)
ระบบนี้พัฒนาขึ้นในรูปแบบ Web Application เพื่อแก้ปัญหาความผิดพลาดของมนุษย์ (Human Error) ในการคำนวณค่าน้ำ ค่าไฟ และยอดรวมค่าเช่าห้องพักแต่ละเดือน นอกจากนี้ยังช่วยอำนวยความสะดวกให้ผู้เช่าสามารถเข้าสู่ระบบมาตรวจสอบยอดค้างชำระ รายละเอียดบิล และ "วันที่กำหนดชำระ" (Due Date) ได้ด้วยตัวเองผ่านอุปกรณ์ใดก็ได้

### 1.2 ขอบเขตของระบบ (Project Scope)
* **สิ่งที่ระบบทำได้:** คำนวณหน่วยน้ำ-ไฟอัตโนมัติจากเลขมิเตอร์, ออกบิลสรุปยอดรวมแต่ละเดือน, รองรับการเข้าสู่ระบบผ่าน Google Account (สำหรับผู้เช่า), และแสดงสถานะการจ่ายเงินพร้อมวันครบกำหนดชำระ
* **สิ่งที่ระบบไม่ได้ครอบคลุม:** ระบบไม่ได้ครอบคลุมถึงการตัดบัตรเครดิตหรือ Payment Gateway เต็มรูปแบบ (ผู้เช่าอาจจะใช้วิธีโอนเงินแล้วอัปโหลดสลิป หรือจ่ายเงินสด แล้วให้ผู้ดูแลมากดอัปเดตสถานะในระบบแทน)

### 1.3 กลุ่มผู้ใช้งาน (User Roles)
1. **Tenant (ผู้เช่า):** ผู้ใช้งานทั่วไปที่เข้าสู่ระบบผ่าน Google Account เพื่อดูบิลค่าห้องของตนเอง
2. **Admin (เจ้าของหอพัก / ผู้ดูแล):** ผู้จัดการระบบที่มีสิทธิ์ตั้งค่าราคาต่อหน่วย กรอกเลขมิเตอร์ และตรวจสอบภาพรวมของทุกห้อง

## 2. ข้อกำหนดการทำงาน (Functional Requirements)

### 2.1 ระบบยืนยันตัวตนและการจัดการสิทธิ์ (Authentication & Authorization)
* `REQ-F-01`: ระบบต้องรองรับให้ผู้เช่าเข้าสู่ระบบ (Log in) ผ่าน **Google Account (Google OAuth)** ได้ เพื่อความสะดวกและไม่ต้องจำรหัสผ่านใหม่
* `REQ-F-02`: แอดมินต้องสามารถผูก "อีเมล Google ของผู้เช่า" เข้ากับ "หมายเลขห้อง" ได้ เพื่อให้ผู้เช่าเห็นข้อมูลเฉพาะห้องของตนเองเมื่อล็อกอินเข้ามา

### 2.2 ระบบคำนวณและลดข้อผิดพลาด (Automated Calculation)
* `REQ-F-03`: แอดมินสามารถตั้งค่าพื้นฐานของหอพักได้ เช่น ค่าห้องพักรายเดือน, เรทค่าน้ำ (บาท/หน่วย), และเรทค่าไฟ (บาท/หน่วย)
* `REQ-F-04`: แอดมินเพียงแค่กรอก **"เลขมิเตอร์น้ำ/ไฟ ล่าสุด"** ของแต่ละเดือน ระบบจะต้องนำไปลบกับเลขมิเตอร์เดือนก่อนหน้าเพื่อหา "จำนวนหน่วยที่ใช้" โดยอัตโนมัติ
* `REQ-F-05`: ระบบต้องคำนวณยอดเงินสุทธิอัตโนมัติ ตามสูตร: `ค่าห้อง + (หน่วยน้ำที่ใช้ x เรทค่าน้ำ) + (หน่วยไฟที่ใช้ x เรทค่าไฟ) + ค่าปรับ/ค่าส่วนกลาง (ถ้ามี)`

### 2.3 ระบบการแสดงผลสำหรับผู้เช่า (Tenant Dashboard)
* `REQ-F-06`: เมื่อผู้เช่าล็อกอินเข้ามา หน้าแรกต้องแสดงบิลของเดือนปัจจุบันอย่างชัดเจน โดยแยกรายละเอียดเป็น ค่าห้อง, ค่าน้ำ, ค่าไฟ และยอดรวม
* `REQ-F-07`: ระบบต้องแสดง **"วันที่ครบกำหนดชำระ (Due Date)"** ตัวใหญ่และชัดเจน (เช่น ภายในวันที่ 5 ของทุกเดือน)
* `REQ-F-08`: ระบบต้องแสดงสถานะบิลของเดือนนั้นๆ เช่น "ยังไม่ชำระ" (Unpaid), "รอตรวจสอบ" (Pending), หรือ "ชำระแล้ว" (Paid)
* `REQ-F-09`: ระบบต้องมีปุ่มให้ผู้เช่าอัปโหลดรูปภาพสลิปโอนเงิน เพื่อแจ้งชำระเงินให้แอดมินทราบ

### 2.4 ระบบหลังบ้านสำหรับแอดมิน (Admin Dashboard)
* `REQ-F-10`: แอดมินต้องเห็นภาพรวมทุกห้องว่า ห้องไหนยังไม่จ่ายบิล เพื่อติดตามทวงถามได้ง่าย
* `REQ-F-11`: แอดมินสามารถกดยืนยันการชำระเงิน เมื่อตรวจสอบสลิปโอนเงินเรียบร้อยแล้ว เพื่อเปลี่ยนสถานะบิลของห้องนั้นเป็น "ชำระแล้ว"

### 2.5 ระบบแจ้งเตือนและฟีเจอร์เพิ่มเติม (Notifications & Enhancements)
* `REQ-F-12`: **LINE Notify Integration:** ระบบต้องสามารถเชื่อมต่อกับ LINE API เพื่อส่งการแจ้งเตือนอัตโนมัติไปยังผู้เช่าเมื่อมีการออกบิลใหม่ หรือใกล้ถึงวันกำหนดชำระ
* `REQ-F-13`: **ประวัติการทำรายการ (Billing History):** ผู้เช่าต้องสามารถดูประวัติบิลย้อนหลังได้อย่างน้อย 12 เดือน
* `REQ-F-14`: **การคิดค่าปรับล่าช้า (Late Fee Calculation):** แอดมินสามารถเปิด/ปิด โหมดคิดค่าปรับอัตโนมัติได้ หากผู้เช่าชำระเกินวันที่กำหนด
* `REQ-F-15`: **การส่งออกข้อมูล (Data Export):** แอดมินต้องสามารถดาวน์โหลดรายงานสรุปยอดขายรายเดือนออกมาเป็นไฟล์นามสกุล `.csv` หรือ `.xlsx` ได้

## 3. ข้อกำหนดที่ไม่เกี่ยวกับการทำงาน (Non-Functional Requirements)

### 3.1 การออกแบบและการใช้งาน (Usability & Responsive)
* **Mobile-First Design:** หน้าเว็บต้องถูกออกแบบให้แสดงผลบนสมาร์ทโฟนได้อย่างสมบูรณ์แบบ
* **ความแม่นยำ (Accuracy):** ตัวแปรที่ใช้เก็บค่าเงินต้องรองรับทศนิยม 2 ตำแหน่ง และมีการปัดเศษที่ถูกต้องตามหลักคณิตศาสตร์

### 3.2 ความปลอดภัยและข้อมูลส่วนบุคคล (Security & Privacy)
* **Data Isolation:** ข้อมูลบิลต้องแยกกันอย่างเด็ดขาด ผู้เช่าต้องไม่สามารถเข้าถึงข้อมูลของห้องอื่นได้
* **Secure Login:** การล็อกอินผ่าน Google ต้องใช้ Token ที่ปลอดภัยและไม่มีการเก็บรหัสผ่าน Google ของผู้ใช้งานไว้ในฐานข้อมูล

## 4. สถาปัตยกรรมระบบและเทคโนโลยี (System Architecture)

### 4.1 เทคโนโลยีฝั่งหน้าบ้าน (Frontend)
* **Framework:** **React.js**
* **รายละเอียด:** ใช้สำหรับการสร้าง User Interface (UI) แบบ Single Page Application (SPA) พัฒนาด้วยแนวคิด Mobile-First

### 4.2 เทคโนโลยีฝั่งหลังบ้าน (Backend / API)
* **Framework:** **Elysia.js** (ทำงานบน Bun runtime)
* **รายละเอียด:** ทำหน้าที่เป็น RESTful API จัดการตรรกะทั้งหมดของระบบ ด้วยความเร็วและประสิทธิภาพที่สูงของ Bun

### 4.3 ระบบฐานข้อมูล (Database)
* **Database Engine:** **MySQL** (Relational Database)
* **รายละเอียด:** ใช้เก็บข้อมูลที่มีโครงสร้างชัดเจนและมีความสัมพันธ์กัน เช่น ข้อมูลผู้ใช้ ข้อมูลมิเตอร์ และประวัติบิล

### 4.4 การติดตั้งและการให้บริการ (Hosting & Deployment)
* **Platform:** **Vercel**
* **รายละเอียด:** นำ Frontend (React.js) ไป Deploy บน Vercel เพื่อใช้ประโยชน์จาก CDN ระดับโลก ส่วน Backend และ Database สามารถเชื่อมต่อผ่านบริการ Hosting ที่รองรับได้
