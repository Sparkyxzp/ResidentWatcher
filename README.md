# Software Requirements Specification (SRS)
**ชื่อโปรเจกต์:** DormBill (ระบบจัดการและตรวจสอบค่าเช่าหอพักออนไลน์)

## 1. บทนำ (Introduction)

### 1.1 จุดประสงค์ (Purpose)
ระบบนี้พัฒนาขึ้นในรูปแบบ Web Application เพื่อแก้ปัญหาความผิดพลาดของมนุษย์ (Human Error) ในการคำนวณค่าน้ำ ค่าไฟ และยอดรวมค่าเช่าห้องพักแต่ละเดือน นอกจากนี้ยังช่วยอำนวยความสะดวกให้ผู้เช่าสามารถเข้าสู่ระบบมาตรวจสอบยอดค้างชำระ รายละเอียดบิล และ "วันที่กำหนดชำระ" (Due Date) ได้ด้วยตัวเอง

### 1.2 ขอบเขตของระบบ (Project Scope)
* **สิ่งที่ระบบทำได้:** มีระบบสมัครสมาชิกและรอแอดมินอนุมัติ, คำนวณหน่วยน้ำ-ไฟอัตโนมัติจากเลขมิเตอร์, ออกบิลสรุปยอดรวมแต่ละเดือน, และแสดงสถานะการจ่ายเงินพร้อมวันครบกำหนดชำระ
* **สิ่งที่ระบบไม่ได้ครอบคลุม:** ระบบไม่ได้ครอบคลุมถึงการตัดบัตรเครดิตหรือ Payment Gateway เต็มรูปแบบ

### 1.3 กลุ่มผู้ใช้งาน (User Roles)
1. **Tenant (ผู้เช่า):** ผู้ใช้งานทั่วไปที่ต้องสมัครสมาชิกและได้รับการอนุมัติจากแอดมินก่อน จึงจะเข้าสู่ระบบเพื่อดูบิลค่าห้องของตนเองได้
2. **Admin (เจ้าของหอพัก / ผู้ดูแล):** ผู้จัดการระบบที่มีสิทธิ์อนุมัติบัญชีผู้ใช้ใหม่ ตั้งค่าราคาต่อหน่วย กรอกเลขมิเตอร์ และตรวจสอบภาพรวมของทุกห้อง

## 2. ข้อกำหนดการทำงาน (Functional Requirements)

### 2.1 ระบบยืนยันตัวตนและการจัดการสิทธิ์ (Authentication & Authorization)
* `REQ-F-01`: **หน้า Login:** ระบบต้องมีหน้าต่างเข้าสู่ระบบเป็นหน้าแรก (Landing Page) หากผู้ใช้ยังไม่มีบัญชี ต้องมีปุ่มให้กดไปที่หน้า "Register"
* `REQ-F-02`: **การสมัครสมาชิก (Register):** ผู้ใช้ใหม่ต้องสามารถกรอกข้อมูลสมัครสมาชิกได้ โดยประกอบด้วย Email, Username, Password และ **เลขห้องพัก** 
* `REQ-F-03`: **สถานะรออนุมัติ (Pending Approval):** หลังจากผู้ใช้กดสมัครสมาชิก บัญชีจะถูกสร้างขึ้นในสถานะ "รอตรวจสอบ" (Pending) โดยที่ผู้ใช้จะยังไม่สามารถ Login เข้าสู่ระบบได้จนกว่าแอดมินจะอนุมัติ
* `REQ-F-04`: **การอนุมัติบัญชี (Account Verification):** ฝั่ง Admin ต้องมีหน้ารายการบัญชีที่รออนุมัติ เพื่อตรวจสอบว่าเป็นลูกบ้านในห้องนั้นจริงๆ หากถูกต้อง Admin สามารถกด "อนุมัติ" (Approve) ได้ หากไม่ใช่สามารถกด "ปฏิเสธ/ลบ" (Reject) ได้
* `REQ-F-05`: **การเข้าสู่ระบบ (Login):** ผู้ใช้ที่ได้รับการอนุมัติจากแอดมินแล้วเท่านั้น จึงจะสามารถใช้ Username (หรือ Email) และ Password เพื่อ Login เข้าสู่ระบบได้

### 2.2 ระบบคำนวณและลดข้อผิดพลาด (Automated Calculation)
* `REQ-F-06`: แอดมินสามารถตั้งค่าพื้นฐานของหอพักได้ เช่น ค่าห้องพักรายเดือน, เรทค่าน้ำ (บาท/หน่วย), และเรทค่าไฟ (บาท/หน่วย)
* `REQ-F-07`: แอดมินเพียงแค่กรอก **"เลขมิเตอร์น้ำ/ไฟ ล่าสุด"** ของแต่ละเดือน ระบบจะต้องนำไปลบกับเลขมิเตอร์เดือนก่อนหน้าเพื่อหา "จำนวนหน่วยที่ใช้" โดยอัตโนมัติ
* `REQ-F-08`: ระบบต้องคำนวณยอดเงินสุทธิอัตโนมัติ ตามสูตร: `ค่าห้อง + (หน่วยน้ำที่ใช้ x เรทค่าน้ำ) + (หน่วยไฟที่ใช้ x เรทค่าไฟ) + ค่าปรับ/ค่าส่วนกลาง (ถ้ามี)`

### 2.3 ระบบการแสดงผลสำหรับผู้เช่า (Tenant Dashboard)
* `REQ-F-09`: เมื่อผู้เช่าล็อกอินเข้ามา หน้าแรกต้องแสดงบิลของเดือนปัจจุบันอย่างชัดเจน โดยแยกรายละเอียดเป็น ค่าห้อง, ค่าน้ำ, ค่าไฟ และยอดรวม
* `REQ-F-10`: ระบบต้องแสดง **"วันที่ครบกำหนดชำระ (Due Date)"** ตัวใหญ่และชัดเจน
* `REQ-F-11`: ระบบต้องแสดงสถานะบิลของเดือนนั้นๆ เช่น "ยังไม่ชำระ" (Unpaid), "รอตรวจสอบ" (Pending), หรือ "ชำระแล้ว" (Paid)
* `REQ-F-12`: ระบบต้องมีปุ่มให้ผู้เช่าอัปโหลดรูปภาพสลิปโอนเงิน เพื่อแจ้งชำระเงินให้แอดมินทราบ

### 2.4 ระบบหลังบ้านสำหรับแอดมิน (Admin Dashboard)
* `REQ-F-13`: แอดมินต้องเห็นภาพรวมทุกห้องว่า ห้องไหนยังไม่จ่ายบิล เพื่อติดตามทวงถามได้ง่าย
* `REQ-F-14`: แอดมินสามารถกดยืนยันการชำระเงิน เมื่อตรวจสอบสลิปโอนเงินเรียบร้อยแล้ว เพื่อเปลี่ยนสถานะบิลของห้องนั้นเป็น "ชำระแล้ว"

### 2.5 ระบบแจ้งเตือนและฟีเจอร์เพิ่มเติม (Notifications & Enhancements)
* `REQ-F-15`: **ประวัติการทำรายการ (Billing History):** ผู้เช่าต้องสามารถดูประวัติบิลย้อนหลังได้อย่างน้อย 12 เดือน

## 3. ข้อกำหนดที่ไม่เกี่ยวกับการทำงาน (Non-Functional Requirements)

### 3.1 การออกแบบและการใช้งาน (Usability & Responsive)
* **Mobile-First Design:** หน้าเว็บต้องถูกออกแบบให้แสดงผลบนสมาร์ทโฟนได้อย่างสมบูรณ์แบบ
* **ความแม่นยำ (Accuracy):** ตัวแปรที่ใช้เก็บค่าเงินต้องรองรับทศนิยม 2 ตำแหน่ง และมีการปัดเศษที่ถูกต้องตามหลักคณิตศาสตร์

### 3.2 ความปลอดภัยและข้อมูลส่วนบุคคล (Security & Privacy)
* **Data Isolation:** ข้อมูลบิลต้องแยกกันอย่างเด็ดขาด ผู้เช่าต้องไม่สามารถเข้าถึงข้อมูลของห้องอื่นได้
* **Password Security:** รหัสผ่านของผู้ใช้จะต้องถูกเข้ารหัส (Hashing) เช่นการใช้ `bcrypt` ก่อนบันทึกลงฐานข้อมูล MySQL เสมอ และไม่มีการเก็บรหัสผ่านเป็น Plain-text เด็ดขาด
* **Access Control:** ทุกๆ API Endpoint ของระบบหลังบ้าน (Elysia) จะต้องมีการตรวจสอบ Token (เช่น JWT) เพื่อยืนยันว่าผู้ใช้ได้ผ่านการ Login และได้รับการอนุมัติแล้วจริงๆ ก่อนส่งข้อมูลให้

## 4. สถาปัตยกรรมระบบและเทคโนโลยี (System Architecture)

### 4.1 เทคโนโลยีฝั่งหน้าบ้าน (Frontend)
* **Framework:** **React.js**
* **รายละเอียด:** ใช้สำหรับการสร้าง User Interface (UI) แบบ Single Page Application (SPA) พัฒนาด้วยแนวคิด Mobile-First

### 4.2 เทคโนโลยีฝั่งหลังบ้าน (Backend / API)
* **Framework:** **Elysia.js** (ทำงานบน Bun runtime)
* **รายละเอียด:** ทำหน้าที่เป็น RESTful API จัดการตรรกะทั้งหมดของระบบ ด้วยความเร็วและประสิทธิภาพที่สูงของ Bun

### 4.3 ระบบฐานข้อมูล (Database)
* **Database Engine:** **MySQL** (Relational Database)
* **รายละเอียด:** ใช้เก็บข้อมูลผู้ใช้ (พร้อมสถานะ Pending/Approved), ข้อมูลมิเตอร์, และประวัติบิล

### 4.4 การติดตั้งและการให้บริการ (Hosting & Deployment)
* **Platform:** **Vercel**
* **รายละเอียด:** นำ Frontend (React.js) ไป Deploy บน Vercel ส่วน Backend (Elysia) และ Database (MySQL) จะนำไป Host บน Platform ที่รองรับตามความเหมาะสม
