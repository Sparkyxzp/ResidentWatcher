= Software Requirements Specification (SRS)
:toc: macro
:toc-title: สารบัญ

*ชื่อโปรเจกต์:* DormBill (ระบบจัดการและตรวจสอบค่าเช่าหอพักออนไลน์)

== 1. บทนำ (Introduction)

=== 1.1 จุดประสงค์ (Purpose)
ระบบนี้พัฒนาขึ้นในรูปแบบ Web Application เพื่อแก้ปัญหาความผิดพลาดของมนุษย์ (Human Error) ในการคำนวณค่าน้ำ ค่าไฟ และยอดรวมค่าเช่าห้องพักแต่ละเดือน นอกจากนี้ยังช่วยอำนวยความสะดวกให้ผู้เช่าสามารถเข้าสู่ระบบมาตรวจสอบยอดค้างชำระ รายละเอียดบิล และ "วันที่กำหนดชำระ" (Due Date) ได้ด้วยตัวเอง

=== 1.2 ขอบเขตของระบบ (Project Scope)
* *สิ่งที่ระบบทำได้:* มีระบบสมัครสมาชิกและรอแอดมินอนุมัติ, คำนวณหน่วยน้ำ-ไฟอัตโนมัติจากเลขมิเตอร์, ออกบิลสรุปยอดรวมแต่ละเดือน, และแสดงสถานะการจ่ายเงินพร้อมวันครบกำหนดชำระ
* *สิ่งที่ระบบไม่ได้ครอบคลุม:* ระบบไม่ได้ครอบคลุมถึงการตัดบัตรเครดิตหรือ Payment Gateway เต็มรูปแบบ (ผู้เช่าใช้วิธีโอนเงินแล้วอัปโหลดสลิป)

=== 1.3 กลุ่มผู้ใช้งาน (User Roles)
1. *Tenant (ผู้เช่า):* ผู้ใช้งานทั่วไปที่ต้องสมัครสมาชิกและได้รับการอนุมัติจากแอดมินก่อน จึงจะเข้าสู่ระบบเพื่อดูบิลค่าห้องของตนเองได้
2. *Admin (เจ้าของหอพัก / ผู้ดูแล):* ผู้จัดการระบบที่มีสิทธิ์อนุมัติบัญชีผู้ใช้ใหม่ ตั้งค่าราคาต่อหน่วย กรอกเลขมิเตอร์ และตรวจสอบภาพรวมของทุกห้อง

== 2. ข้อกำหนดการทำงาน (Functional Requirements)

=== 2.1 หน้าเข้าสู่ระบบและลงทะเบียน (Public / Shared Gate)
* `REQ-PUB-01`: *หน้า Login:* ระบบต้องแสดงฟอร์มเข้าสู่ระบบเป็นหน้าแรก หากกรอกบัญชีที่ได้รับการอนุมัติแล้ว ระบบจะส่งผู้ใช้ไปยังหน้า Dashboard ตามสิทธิ์ (Tenant หรือ Admin)
* `REQ-PUB-02`: *หน้า Register (สำหรับผู้เช่าใหม่):* ต้องมีฟอร์มให้กรอก Email, Username, Password และ *เลขห้องพัก* เมื่อกดปุ่มสมัคร ระบบจะบันทึกข้อมูลและตั้งสถานะเป็น "รอตรวจสอบ" (Pending) พร้อมแสดงข้อความเตือนให้ผู้ใช้รอแอดมินอนุมัติ

=== 2.2 ระบบฝั่งผู้เช่า (Tenant / User Side)
* `REQ-TEN-01`: *ระบบควบคุมการเข้าถึง (Access Control):* หากบัญชียังอยู่ในสถานะ "รอตรวจสอบ" (Pending) ระบบต้องบล็อกไม่ให้เข้าถึงหน้าหลัก และแสดงหน้าจอแจ้งเตือน "กำลังรอการอนุมัติจากผู้ดูแลหอพัก"
* `REQ-TEN-02`: *แดชบอร์ดสรุปยอด (Tenant Dashboard):* เมื่อเข้าสู่ระบบสำเร็จ หน้าแรกต้องแสดงบิลค่าเช่าของเดือนปัจจุบัน โดยแยกรายละเอียดชัดเจน: ยอดรวมสุทธิที่ต้องชำระ, วันกำหนดชำระ (Due Date), สถานะของบิล (ยังไม่ชำระ / รอตรวจสอบสลิป / ชำระแล้ว), และรายละเอียดค่าใช้จ่ายย่อย (ค่าห้องพื้นฐาน, ค่าน้ำ, ค่าไฟ)
* `REQ-TEN-03`: *การแจ้งชำระเงิน (Payment Submission):* มีปุ่มและช่องทางให้ผู้เช่าสามารถอัปโหลดรูปภาพสลิปโอนเงิน (Slip Verification) เพื่อส่งให้แอดมินตรวจสอบ และระบบจะเปลี่ยนสถานะบิลอัตโนมัติเป็น "รอตรวจสอบ"
* `REQ-TEN-04`: *ประวัติบิลย้อนหลัง (Billing History):* มีเมนูให้ผู้เช่าเลือกดูและดาวน์โหลดบิลค่าเช่าย้อนหลังของห้องตนเองได้สูงสุด 12 เดือน

=== 2.3 ระบบฝั่งแอดมิน (Admin / Back-Office Side)
* `REQ-ADM-01`: *ระบบอนุมัติลูกบ้านใหม่ (Tenant Approval Panel):* มีหน้าจอแสดงรายชื่อผู้เช่าที่สมัครสมาชิกเข้ามาใหม่ (สถานะ Pending) แสดงพร้อม Email, Username และเลขห้อง โดยแอดมินมีสิทธิ์กด "อนุมัติ (Approve)" หรือ "ปฏิเสธ (Reject)"
* `REQ-ADM-02`: *แดชบอร์ดภาพรวมหอพัก (Admin Dashboard):* หน้าจอสรุปสถานะของทุกห้องพักในหอพักแบบ Real-time (เช่น สีแดง = ยังไม่จ่าย, สีเหลือง = รอตรวจสลิป, สีเขียว = จ่ายแล้ว)
* `REQ-ADM-03`: *ระบบบันทึกมิเตอร์น้ำ-ไฟ (Meter Recording System):* แอดมินสามารถเลือกทีละห้องเพื่อกรอก "เลขมิเตอร์น้ำ" และ "เลขมิเตอร์ไฟ" ล่าสุดของเดือนนั้นๆ โดยระบบจะคำนวณจำนวนหน่วยที่ใช้และยอดเงินให้อัตโนมัติ
* `REQ-ADM-04`: *การตรวจสอบและยืนยันยอดเงิน (Payment Verification):* หน้าจอสำหรับตรวจรูปภาพสลิปโอนเงินที่ผู้เช่าส่งมา แอดมินสามารถกดอนุมัติเพื่อเปลี่ยนสถานะเป็นชำระแล้ว หรือปฏิเสธได้
* `REQ-ADM-05`: *การตั้งค่าระบบ (System Settings):* แอดมินสามารถแก้ไขค่าใช้จ่ายส่วนกลาง เรทค่าน้ำ/ค่าไฟต่อหน่วย และกำหนดวัน Due Date ได้
* `REQ-ADM-06`: *การส่งออกรายงาน (Data Export):* แอดมินสามารถกดส่งออกรายงานสรุปรายรับประจำเดือนออกมาเป็นไฟล์ Excel หรือ CSV ได้

== 3. ข้อกำหนดที่ไม่เกี่ยวกับการทำงาน (Non-Functional Requirements)

=== 3.1 การออกแบบและการใช้งาน (Usability & Responsive)
* *Mobile-First Design:* หน้าเว็บต้องถูกออกแบบให้แสดงผลบนสมาร์ทโฟนได้อย่างสมบูรณ์แบบ
* *ความแม่นยำ (Accuracy):* ตัวแปรที่ใช้เก็บค่าเงินต้องรองรับทศนิยม 2 ตำแหน่ง และมีการปัดเศษที่ถูกต้องตามหลักคณิตศาสตร์

=== 3.2 ความปลอดภัยและข้อมูลส่วนบุคคล (Security & Privacy)
* *Data Isolation:* ข้อมูลบิลต้องแยกกันอย่างเด็ดขาด ผู้เช่าต้องไม่สามารถเข้าถึงข้อมูลของห้องอื่นได้
* *Password Security:* รหัสผ่านของผู้ใช้จะต้องถูกเข้ารหัส (Hashing) เช่นการใช้ `bcrypt` ก่อนบันทึกลงฐานข้อมูล MySQL เสมอ
* *Access Control:* ทุกๆ API Endpoint ของระบบหลังบ้าน (Elysia) จะต้องมีการตรวจสอบ Token (เช่น JWT)

---

== 4. สถาปัตยกรรมระบบและเทคโนโลยี (System Architecture)

=== 4.1 เทคโนโลยีฝั่งหน้าบ้าน (Frontend)
* *Framework:* *React.js*
* *รายละเอียด:* ใช้สำหรับการสร้าง User Interface (UI) แบบ Single Page Application (SPA) พัฒนาด้วยแนวคิด Mobile-First

=== 4.2 เทคโนโลยีฝั่งหลังบ้าน (Backend / API)
* *Framework:* *Elysia.js* (ทำงานบน Bun runtime)
* *รายละเอียด:* ทำหน้าที่เป็น RESTful API จัดการตรรกะทั้งหมดของระบบ ด้วยความเร็วและประสิทธิภาพที่สูงของ Bun

=== 4.3 ระบบฐานข้อมูล (Database)
* *Database Engine:* *MySQL* (Relational Database)
* *รายละเอียด:* ใช้เก็บข้อมูลผู้ใช้ (พร้อมสถานะ Pending/Approved), ข้อมูลมิเตอร์, และประวัติบิล

=== 4.4 การติดตั้งและการให้บริการ (Hosting & Deployment)
* *Platform:* *Vercel*
* *รายละเอียด:* นำ Frontend (React.js) ไป Deploy บน Vercel ส่วน Backend (Elysia) และ Database (MySQL) จะนำไป Host บน Platform ที่รองรับตามความเหมาะสม

== 8. ข้อสมมติฐานและข้อจำกัดของระบบ (Assumptions & Dependencies)

=== 8.1 ข้อสมมติฐาน (Assumptions)
* ผู้เช่าทุกคนต้องมีอุปกรณ์ที่สามารถเข้าถึงอินเทอร์เน็ตได้เพื่อใช้ในการตรวจสอบบิลและแจ้งชำระเงิน
* แอดมินเป็นผู้ตรวจสอบและบันทึกเลขมิเตอร์ที่อ่านได้จากหน้าห้องพักจริงเข้าสู่ระบบอย่างถูกต้อง

=== 8.2 การพึ่งพาภายนอก (Dependencies)
* ระบบต้องพึ่งพาเสถียรภาพของเซิร์ฟเวอร์ MySQL และการเชื่อมต่อเครือข่ายของ Vercel ในการรับส่งข้อมูล API
* ความปลอดภัยของรหัสผ่านขึ้นอยู่กับกระบวนการเข้ารหัสผ่านคำสั่ง Hashing ลับฝั่ง Backend (Elysia.js)

== 10. ข้อกำหนดการเชื่อมต่อ API (API Interface Specifications)

สถาปัตยกรรม RESTful API พื้นฐานสำหรับการสื่อสารระหว่าง Frontend (React) และ Backend (Elysia):

* `POST /api/auth/register`: สำหรับรับข้อมูลสมัครสมาชิกใหม่ (สถานะเริ่มต้นจะเป็น Pending)
* `POST /api/auth/login`: สำหรับตรวจสอบสิทธิ์และรับ Token (JWT) เพื่อเข้าใช้งานระบบ
* `GET /api/invoices/current`: สำหรับดึงข้อมูลบิลค่าน้ำค่าไฟของเดือนปัจจุบัน (ผู้ใช้ต้องมี Token แนบมาด้วย)
* `POST /api/payments/upload`: สำหรับอัปโหลดรูปภาพสลิปโอนเงิน (รองรับ multipart/form-data)
* `PUT /api/admin/users/{id}/approve`: (สิทธิ์ Admin เท่านั้น) สำหรับอนุมัติให้บัญชีผู้ใช้งานสามารถเข้าสู่ระบบได้

== 12. เกณฑ์การยอมรับระบบ (Acceptance Criteria / UAT)

ระบบจะถือว่าพัฒนาเสร็จสมบูรณ์เมื่อผ่านการทดสอบ User Acceptance Testing ในหัวข้อหลักดังนี้:
1. *ความถูกต้องของบัญชี:* ผู้เช่าที่สมัครเข้ามาใหม่จะต้องไม่สามารถ Login ได้เด็ดขาดจนกว่าแอดมินจะกด Approve
2. *การคำนวณยอดสุทธิ:* ระบบต้องคำนวณยอดบิล (ค่าห้อง + ค่าน้ำ + ค่าไฟ) ได้ถูกต้อง 100% ไม่มีเศษทศนิยมผิดพลาดเมื่อเทียบกับการคำนวณด้วยมือ
3. *การอัปโหลดไฟล์:* ผู้เช่าสามารถอัปโหลดไฟล์ภาพ (.jpg, .png) สลิปโอนเงินได้ และแอดมินสามารถเปิดดูภาพนั้นได้ชัดเจน
4. *ความปลอดภัย:* ระบบจะต้องป้องกันไม่ให้ผู้ใช้งานระดับ Tenant สามารถเรียกดูข้อมูลผ่าน URL ของหน้า Admin ได้ (ต้อง Redirect กลับไปหน้าหลัก)

== 13. อภิธานศัพท์ (Glossary)

* *Due Date:* วันกำหนดชำระเงินวันสุดท้ายของแต่ละรอบบิล
* *Pending:* สถานะรอดำเนินการ แบ่งเป็น 2 กรณีคือ 1. บัญชีรอการอนุมัติ 2. บิลที่โอนเงินแล้วแต่รอแอดมินตรวจสลิป
* *Tenant:* ผู้เช่า หรือลูกบ้าน ที่อาศัยอยู่ในหอพัก
* *Approve / Reject:* การอนุมัติ / การปฏิเสธคำขอ (ใช้ในการจัดการบัญชีผู้ใช้และการตรวจสลิปโอนเงิน)
